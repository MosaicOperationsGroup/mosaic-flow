The rulebook

The orchestration layer

Roles, quality gates and durable records that make every piece of work traceable and provable — whoever did it.

Every session starts with one command pointed at a real issue: /mosaic-dispatch 121. It reads the issue, checks the map of who owns that area of the product, classifies how risky the change is (T0 to T4), and picks a route.

Example: issue #121, "add archive to tasks", touches task management → classified tier T2 → the standard route → intake, brief, implement, verify, close — each stage leaving a real file behind in the repository, not just a chat message.

The flow

From backlog to proven, in nine moves.

Work isn't just done — it's planned, bounded, independently checked, and recorded. Rigor scales with risk: a colour tweak takes the short path, a data migration takes every step.

0

Session

Human + orchestrator

The tracker is triaged into a few clear goal options. A human picks one and approves the plan.

The orchestrator agent groups open issues into session-sized goals and drafts a resourcing plan — which role works each step, on which model, at what effort, in what order. Nothing starts until the plan is approved. The plan is a pre-commitment: what actually ran is recorded separately, so drift is visible.

1

Dispatch

Orchestrator

Every issue gets a risk tier, T0 to T4, and a route to match.

Anything touching migrations, permissions, or personal data is forced up the tiers automatically. Higher tier means more scrutiny — and a record is opened in the ledger from the very start, even for trivial fixes.

2

Intake

Context scout

Before anything is built, an agent reads the governing docs — and must cite them.

Product specs, architecture decisions, prior rulings. Work that can't cite its sources is incomplete by definition. This exists because building on assumptions is the most common way AI work goes wrong.

3

Brief

Brief compiler

A self-contained work order: exact scope, success criteria, checks to run, and when to stop and ask.

The brief is written so that a fresh session — on any of the three runtimes — can execute it with zero prior context. Assumptions are numbered and surfaced, never silently made. Success criteria are written before the code.

4

Implement

Implementer

Build exactly what the brief asks. Nothing more.

The implementer may only touch the files the brief allows. Every changed line must trace to the request — no drive-by refactors, no gold-plating. It runs the checks, reviews its own diff, and hands over an honest receipt of what it did and what passed.

5

Verify

Verifier — fresh eyes

A different agent, deliberately shown nothing of the author's reasoning, tries to break it.

It re-runs every check itself and tests each success criterion adversarially. For the riskiest work, the verifier must be a different vendor's model entirely — different blind spots — plus a deep security review, triggered only by a human.

6

Gate

Machine-checked

The verdict is written down — and a script checks the record is complete and honest.

The gate file records the evidence per criterion, who built and who checked (they must differ), and what was skipped or waived — with an owner and an expiry. Continuous checks even warn if a bigger AI model was used than the risk justified.

7

Close

Integrator + scribe

Merged, recorded, and tidied — by the smallest agent that can do the job.

Closing is refused without a passing gate. Then the bookkeeping: progress records updated, follow-ups filed, learnings drafted — mechanical work with a right answer, done by a fast, inexpensive model.

8

Sweep

Sweep reviewer — the safety net

Trivial fixes may close themselves — provisionally. A periodic reviewer re-checks all of them.

Runs on its own every night — nobody has to remember to trigger it. It looks for two things: small fixes an agent already closed itself ("self-closed, awaiting sweep"), and work that looks finished — a merged change, a receipt on file — that nobody independently checked yet.

For each one, it re-reads the acceptance criteria and re-runs the checks itself, from scratch. Genuinely good work gets confirmed and properly closed. Anything that doesn't hold up gets reopened, with a concrete list of what's actually wrong — a sweep that only ever agrees would be worthless, so it's built to disagree when the evidence says so.

Example: a sidebar spacing fix self-closes the moment its own build passes. That night, the sweep re-runs the build itself, confirms nothing regressed, and marks it genuinely verified — not just "probably fine".

Before the pipeline

How a new idea becomes something the pipeline can build.

Most work is a small, well-understood fix — file the issue, and the session above (dispatch onward) picks it up directly. A bigger or riskier feature gets a design pass first, so agents build against an agreed plan instead of a guess.

Small fix? Skip everything below. File the issue, run a session, and it goes straight into dispatch → implement → verify like anything else.
0

Structured intake

Human — the Karpathy intake

A 13-part form turns a rough idea into something two independent agents can build from.

Goal, users, workflows, non-goals, assumptions (with confidence and risk), open decisions, data model implications, permissions and RLS implications, acceptance criteria, a verifier matrix, a seven-question architecture check, proposed implementation issues, and reconciliation notes. Named after Andrej Karpathy's writing on structuring intent for AI agents — the point is surfacing what you assumed but never said out loud.

1

Two blind drafts

Agent A + Agent B — independently

Two agents each write a full spec from the same intake. Neither sees the other's draft.

Independent first drafts surface different blind spots than one agent iterating on its own idea. Either agent can be Claude, Codex, or Gemini — the process doesn't care which, only that they worked apart.

2

Reconcile

Agent A

The first agent reads both drafts, keeps the strongest points from each, and resolves the blocking disagreements.

3

Adversarial response

Agent B

The second agent pushes back — accepting, rejecting, or refining. Only the points that still matter get reopened.

4

Converge

Both agents

More rounds only happen if a real blocker remains — an open security question, a data-model conflict.

It stops once the plan is specific enough to build and every blocking question about data, security, and product rules has a real answer — not once it merely looks finished.

5

Accept & hand off

Human — product owner

The accepted plan becomes the feature's product spec, its architecture decisions, and an implementation brief — then issues get filed and the pipeline above takes over.

Real example: the task and workflow data model went through exactly this — two independent drafts, reconciled, then accepted as an architecture decision, replacing two earlier attempts once adversarial review showed a single shared shape didn't hold up once a task needed to belong to more than one workflow.

Proof before belief

How Mosaic knows a change actually works.

Nothing is believed because it looks right. It's believed because it was checked against a known, memorised fake school — twice: once with shortcuts, once for real.

Two runtimes, on purpose

Preview vs. the real thing

A fast fake login for building quickly, and the real system with real permissions for proving it. Every feature is checked in both.

Preview mode logs you in as a pretend "Dev User", uses made-up data that resets on reload, and skips permission checks entirely — it's for fast iteration, not proof. The authenticated mode is the real deployed system: a real login, a real database, and permissions actually enforced by the server. A difference between the two that nobody wrote down is treated as a bug, full stop.

The golden fixture

One imaginary school, memorised

A cast of specific, named people whose edge cases are the whole point — an admin who can't be assigned tasks, a staff member who's archived but can still log in, someone who belongs to two schools at once.

Plus a second "foil" organisation whose entire job is proving the real one can't see into it. Every count is written down and checked by name: the school has 10 people, but exactly 8 ever appear in an "assign to everyone" list, because two admin accounts aren't allowed to hold tasks. If that number is ever 9, something is broken — and the foil organisation's rows must always be zero, on every screen, for everyone else.

The test oracle

A table that can't lie

Every feature keeps a written table: given this exact starting point, this exact action should produce this exact result — in both runtimes — naming the real test that checks it, or an honest "gap" if nothing does yet.

Real example, task assignment: "assign to all members" must offer exactly 8 people, never the 2 who aren't allowed to hold tasks — checked by tests/task-assignment.spec.ts. A feature can't quietly drift from what was agreed, because the table names the exact number and the exact test.

Never real data

Dev sandbox vs. a live rehearsal

The dev server's "dummy data" is make-believe on one machine. The live site's "dummy accounts" are real accounts, on the real deployed system — just for a school that doesn't exist.

A developer's local preview never touches a real database at all. Testing "for real" means real accounts, a real login, real permission checks — but seeded onto a disposable practice organisation, not a real school. Real student and staff data is never put into any test environment, ever — no exceptions, even this early in development.

Migrations

Treated like surgery, not an edit

Five things must be true before any change touches the real database: a verified backup, a rehearsal on a disposable copy, additive not destructive, before/after counts written down, and a stated recovery plan.

Every database change is tried first on a throwaway copy seeded from the golden fixture — never on the real one. Anything that could destroy real data, or that an AI wrote unreviewed, gets the maximum scrutiny: a deep multi-agent security pass (/code-review ultra — "ultrareview") plus a named human who explicitly signs off before it ships. It's triggered by a person, on purpose — never launched automatically.

Why it matters

A real example

The gradebook feature's first draft would have let every staff member see every student's grades, forever, with no record of where an imported spreadsheet's numbers actually came from.

The adversarial design review (previous section) caught this before a line of code shipped. It was fixed to: access narrowed to each teacher's own classes, imports leave a traceable trail back to their source file, and nothing writes to the database directly — every write goes through a checked, audited door. This is exactly why import and export paths always get extra scrutiny: a file a person uploads is data the system hasn't earned trust in yet.

People & agents

Humans decide. Agents do. Records prove.

People are heaviest at the edges — choosing what matters and signing off what's risky. Agents own the middle, each sized to its job.

Product owner

Human

Picks the goals, approves the plans, triggers deep reviews, and gives the final go / no-go on the riskiest changes.

The owner never has to check everything — the flow is built so their attention lands only where judgment is irreplaceable: direction, risk, and release.

Testers

Human

Real people use the product and log what they find — through an agent, straight into the tracker.

Human findings become labelled issues with priorities, which is exactly where the next session's goals come from. The loop starts with people.

Orchestrator

Judgment

Turns the backlog into goal options and a staffing plan a human can approve in one look.

Read-only until the plan is approved. It assigns models by risk tier — and never spends a bigger mind than the work deserves.

Context scout

Standard

Reads the specs, decisions and code first, so nothing is built on assumptions.

Produces a cited brief of binding constraints and contradictions. Cheap insurance against the expensive failure of building the wrong thing well.

Brief compiler

Standard

Writes the self-contained work order any agent, on any runtime, can execute cold.

Scope, success criteria, checks, stop conditions, numbered assumptions. If two readings of a request differ, it asks — it never guesses.

Implementer

Standard

Builds exactly what the order asks, inside the allowed files, and returns an honest receipt.

Simplicity first, surgical changes only. It reviews its own diff before handoff — then someone else grades it.

Verifier

Judgment

Fresh eyes that try to falsify "it's done" — and write the verdict into the ledger.

Never the author. Re-runs every check itself; evidence it didn't reproduce doesn't count. Its verdict is what unlocks the close.

Security reviewer

Judgment

For high-risk work: probes tenant isolation, permissions and migrations against the test school.

Checks that school B sees zero of school A's rows, that denied paths actually deny, and that no personal data leaks into logs. Probes or it didn't happen.

Sweep reviewer

Judgment

Re-checks self-closed and unverified-but-finished work. Can confirm — or reopen.

The safety net that makes fast paths safe. A sweep that only ever agrees would be worthless, so it's built to reopen.

Scribe

Mechanical

Bookkeeping with a right answer: progress records, follow-up issues, release notes, ledger tidying.

Examples: updating a feature's progress page after a close, filing the follow-up when a waiver expires, drafting the changelog line. No judgment required — so no expensive model used.

Right-sized minds. Judgment roles run on frontier models (Claude Opus, top-effort Codex, Gemini Pro at high thinking). Standard roles run mid-tier (Claude Sonnet, Gemini Pro). Mechanical roles run small and fast (Claude Haiku, Gemini Flash). The ledger records what was actually used — and flags mismatches.

The rules

Four rules no change can skip.

Rule 1

Read before you build

Every piece of work must cite the specs and decisions that govern it. Uncited work is incomplete — because building on assumptions is how systems quietly rot.

Every feature keeps its own folder of governing documents, and an agent starts there — not in the code:

docs/features/<feature>/
  README.md            — start here: status, links, blockers
  product-spec.md      — the problem, the users, what "done" means
  decisions/           — binding architecture & data-model rules
  progress.md          — what's actually built, what's still open
  expected-outcomes.md — the test oracle (see Testing, below)

When sources disagree, the real database and code outrank even an accepted decision record, which outranks the spec, which outranks a rough note. And there's a map from "what you're touching" to "what to read first" — edit task assignment, and the rule says read Roles & Permissions before Task Management, because "who's allowed to hold a task" is a permissions concept, not a task one.

Rule 2

Authors never grade their own work

Anything substantive is verified by fresh eyes that saw none of the author's reasoning. Trivial fixes may close themselves — but only provisionally, until the sweep confirms.

This is a rule about agents, not the humans overseeing them — a product owner can approve their own decisions; what's forbidden is an AI agent being both the builder and the sole judge of its own code.

Example: the agent that adds task archiving doesn't get to decide its own tests pass — a separate agent, shown only the acceptance criteria and not the first agent's reasoning, re-runs everything and tries to break it.

Example: a one-line colour fix may close itself immediately — but it's marked provisional until that night's sweep, run by a third, independent pass, confirms it for real.

Rule 3

Rigor scales with risk

A colour tweak and a data migration are not the same thing. Five risk tiers decide how much scrutiny a change gets — up to cross-vendor review and a human signature.

Rule 4

Everything leaves a record

Plans, receipts, verdicts, waivers — written down in the repo, checked by machines. The question "who verified this, and how?" always has an answer.
